Relational Database Schema Design (10+ Foreign Keys)

users (User Entity)
id : BIGINT [PK] name : VARCHAR(255) email : VARCHAR(255) [UNIQUE] phone : VARCHAR(255) role : VARCHAR(50) [RIDER/DRIVER] rating : DOUBLE balance : DOUBLE
vehicles (Vehicle Entity)
id : BIGINT [PK] driver_id : BIGINT [FK -> users] make : VARCHAR(255) model : VARCHAR(255) license_plate : VARCHAR [UNIQUE] capacity : INT vehicle_type : VARCHAR [SEDAN/SUV...]
driver_availability
id : BIGINT [PK] driver_id : BIGINT [FK -> users, UNIQUE] current_latitude : DOUBLE current_longitude : DOUBLE is_available : BOOLEAN last_updated_at : DATETIME
ride_requests
id : BIGINT [PK] rider_id : BIGINT [FK -> users] start_latitude : DOUBLE start_longitude : DOUBLE end_latitude : DOUBLE end_longitude : DOUBLE requested_type : VARCHAR status : VARCHAR [PENDING/MATCHED...] passenger_count : INT requested_at : DATETIME
rides (Ride Entity)
id : BIGINT [PK] request_id : BIGINT [FK -> requests, UNIQUE] rider_id : BIGINT [FK -> users] driver_id : BIGINT [FK -> users] vehicle_id : BIGINT [FK -> vehicles] status : VARCHAR fare : DOUBLE start_time : DATETIME end_time : DATETIME
payments
id : BIGINT [PK] ride_id : BIGINT [FK -> rides, UNIQUE] payer_id : BIGINT [FK -> users] amount : DOUBLE status : VARCHAR [PENDING/COMPLETED...] payment_method : VARCHAR payment_time : DATETIME
reviews
id : BIGINT [PK] ride_id : BIGINT [FK -> rides] reviewer_id : BIGINT [FK -> users] reviewee_id : BIGINT [FK -> users] rating : INT comment : VARCHAR(500) created_at : DATETIME
ride_match_conflicts
id : BIGINT [PK] request_id : BIGINT [FK -> requests] driver_id : BIGINT [FK -> users, NULL] conflict_type : VARCHAR details : VARCHAR(500) resolution_action : VARCHAR(255) resolved_at : DATETIME

Data Integrity & System Design Highlights

Foreign Key Constraints Implemented (Total: 14)

  1. fk_vehicles_driver: Maintains that every vehicle is driven by a valid user profile.
  2. fk_availability_driver: Assures location records map to registered user entities.
  3. fk_requests_rider: Connects bookings to correct customer entity.
  4. fk_rides_request: Avoids orphan rides by securing 1-to-1 matching with requests.
  5. fk_rides_rider: Double-verifies passenger profiles for security audits.
  6. fk_rides_driver: Solidifies driver record accountability on historical rides.
  7. fk_rides_vehicle: Tracks vehicle usage integrity on ride-matching outcomes.
  8. fk_payments_ride: Locks transaction history mapping directly to verified ride settlements.
  9. fk_payments_payer: Assures debit authorization is tied to matching payer ID.
  10. fk_reviews_ride: Links reviews only to completed and verified rides.
  11. fk_reviews_reviewer: Enforces valid author attributes.
  12. fk_reviews_reviewee: Protects target customer ratings.
  13. fk_conflicts_request: Details diagnostics precisely against the origin request.
  14. fk_conflicts_driver: Connects driver scheduling constraints to logs.

Object-Oriented Design Principles Used

  • Polymorphism / Roles: Unified User table acting as Riders and Drivers, configured dynamically.
  • Encapsulation: Repository layer abstracts data operations completely from business models.
  • Conflict Strategy Design: Modular, robust services implementing real-time algorithms (Distance Proximity, Capacity Filtration, Concurrent Safety).
